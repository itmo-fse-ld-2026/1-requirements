\subsubsection{Apportioning of Requirements}

Завершая текущую главу, следует отметить, что описанная система обладает рядом спорных архитектурных решений. Перечислим некоторые из них, чтобы сопровождающая сторона Сайта могла внести изменения в будущих версиях проекта.

\subsubsubsubsection{Entrypoints Organization}

Текущая реализация Сайта использует для доступа к ресурсам прямой доступ к файлам, а для выполнения серверной логики --- прямое исполнение PHP-скриптов. Такой формат реализации публичного API нарушает стиль REST \textit{(Representational State Transfer)}. В частности, интерфейс зависит от своей технической реализации из-за суффиксов \textit{.php}. Интерфейс должен быть абстрактным, чтобы в дальнейшем при необходимости можно было безболезненно поменять техническую реализацию.

Более того, следует пересмотреть стиль наименований точек входа. Текущая версия Сайта использует транслитерацию с русского на английский для обозначения названий ресурсов. Большая часть сайтов в российском сегменте сети Интернет всё же использует английские термины в URL. Либо как альтернативу можно рассмотреть переход на IDN \textit{(Internationalized Domain Names)}, чтобы использовать в URL лишь русский алфавит.

\subsubsubsubsection{Form Validation}

Текущая реализация Сайта обладает рядом форм, которые реализуют валидацию своих полей. Однако требования к форме проверяются лишь в момент отправки формы. Из-за этого, в частности, у Пользователя может возникнуть ложное ощущение того, что комментарии могут быть бесконечно большими.

Предлагаем для форм добавить индикаторы, которые будут свидетельстовать о корректности вводимых данных. В частности, для текстовых полей следует внедрить счетчики символов с визуальной индикацией приближения к лимиту. Это исключит ситуацию «неоправданных ожиданий», когда Пользователь вводит объёмный текст, который система не сможет обработать.

\subsubsubsubsection{Comment Pagination}

Текущая реализация Сайта показывает пользователю последние несколько комментариев к странице. Если их много, то появляется кнопка «Смотреть все комментарии», которая берёт из базы данных все записи и предоставляет Пользователю за раз. Из-за этого страница может вырасти бесконечно далеко вниз, что плохо.

В качестве решения предлагаем изменить кнопку на «Смотреть следующие комментарии». Вместо всех комментариев она будет загружать лишь следующую страницу из \( n \) штук. Для этого необходимо расширить API, чтобы он мог отвечать на запросы с указанными параметрами пагинации.

\subsubsubsubsection{DDoS Protection}

Если Администратор совершает несколько неудачных попыток входа в Панель управления, то к нему применится метод блокировки по IP-адресу. В современности этот метод неэффективен, потому что чаще всего IP-адреса динамичны. Более того, в таком сценарии может быть заблокирована группа Пользователей, которая выходит в сеть Интернет через один NAT-шлюз.

Вместо полной блокировки доступа следует активировать механизм CAPTCHA. Это позволит отсечь автоматизированные скрипты, не ограничивая доступ для человека, который мог ошибиться при вводе пароля. Также это устраняет угрозу обхода блокировки через простую ротацию IP-адресов злоумышленником.

\subsubsubsubsection{Local Search Platform}

Использование PSE Google добавляет Сайту зависимость от внешнего сервиса. Более того, Google может продвигать другие результаты поиска, которые в целом не относятся к Сайту. Ещё проблематично визуально интегрировать стороннюю поисковую систему, если оформление Сайта будет пересмотрено в будущем.

Предлагается внедрение локального поискового решения. Для этого можно рассмотреть готовые решения полнотекстового поиска в PostgreSQL.

\subsubsubsubsection{Remove Sessions}

Текущая реализация сохранения «избранных» страниц через сессионные Cookies нецелесообразна. Сайт позиционирует себя как ресурс без обязательной регистрации, поэтому хранение дополнительного списка идентификаторов страниц в сессии заставляет сервер хранить больше данных.

Необходимо перенести логику хранения «избранного» в локальное хранилище веб-браузера. Это позволит убрать хранение пользовательского состояния, которое нигде больше не используется.

\subsubsubsubsection{Comment Draft Auto-Save}

При написании длинного комментария пользователь рискует потерять текст при случайном обновлении страницы или закрытии вкладки.

Предлагается автоматически сохранять черновик комментария в локальном хранилище браузера и восстанавливать его при повторном открытии страницы.

\subsubsubsubsection{Article Categories}

Раздел статей не имеет категоризации. При увеличении количества статей навигация по ним станет неудобной.

Предлагается добавить категории для статей и возможность фильтрации по категориям на странице списка статей.

\subsubsubsubsection{User Account and Profile}

Сайт позиционируется как ресурс без обязательной регистрации, но отсутствие опционального личного кабинета ограничивает возможности персонализации гаданий и сохранения результатов Пользователя.

Предлагается добавить опциональную регистрацию с возможностью ведения личного кабинета. Основной функционал должен оставаться доступным без регистрации.

\subsubsubsubsection{Recently Viewed Items}

Пользователь может забыть добавить интересное гадание в избранное и потом не сможет его найти. Также нет возможности быстро вернуться к недавно просмотренным материалам.

Предлагается добавить блок «Недавно просмотренные» на главной странице или в боковой колонке, который отображает последние 5–10 посещённых страниц. Хранение истории должно быть реализовано в локальном хранилище браузера, без привязки к серверу.
